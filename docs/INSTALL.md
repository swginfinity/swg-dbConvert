# Installation, Build & Usage Guide

Step-by-step instructions for adding `dbconvert` to your SWGEmu server.

## Prerequisites

- A working SWGEmu Core3 build environment (GCC/Clang, CMake, BerkeleyDB 5.3, Boost)
- Your server must build and run successfully before adding dbconvert
- Python 3 (for the autogen patch script)

## Automated Build (Recommended)

The `build.sh` script handles everything automatically — engine3 patches, autogen patches, CMake target, and compilation. It reverts all patches when done, leaving engine3 clean.

```bash
./scripts/build.sh /path/to/MMOCoreORB
```

If you prefer to understand each step (or build.sh doesn't fit your setup), follow the manual steps below.

---

## Manual Build

### Step 1: Back Up Your Databases

**Always back up before conversion.** Conversion modifies databases in place.

```bash
# Simple copy
cp -r MMOCoreORB/bin/databases/ MMOCoreORB/bin/databases_backup_$(date +%Y%m%d)/

# Or use the included backup script
./scripts/backup.sh pre_convert /path/to/MMOCoreORB/bin/databases
```

### Step 2: Add Source Files

```bash
# From your MMOCoreORB directory:

# Create tools directory if it doesn't exist
mkdir -p src/tools

# Copy the converter source
cp /path/to/swgemu-dbconvert/src/tools/dbconvert.cpp src/tools/
```

### Step 3: Apply Engine3 Patches

These 4 patches add guards needed for standalone (non-server) mode. Without them, dbconvert will crash or silently produce incorrect results.

```bash
cd /path/to/MMOCoreORB/utils/engine3

# Apply all 4 patches
git apply /path/to/swgemu-dbconvert/patches/001-taskmanager-standalone-mode.patch
git apply /path/to/swgemu-dbconvert/patches/002-uncompress-error-handling.patch
git apply /path/to/swgemu-dbconvert/patches/003-objectdatabase-decompress-guard.patch
git apply /path/to/swgemu-dbconvert/patches/004-orb-null-check.patch
```

What each patch does:

| # | Patch | File | Purpose |
|---|-------|------|---------|
| 1 | `001-taskmanager-standalone-mode` | `TaskManagerImpl.cpp` | Early return when `workerCount < 0`. Prevents divide-by-zero in standalone mode. |
| 2 | `002-uncompress-error-handling` | `LocalDatabase.cpp`, `LocalDatabase.h` | Changes `uncompress()` from `void` to `bool`. Returns `false` on corrupt data. |
| 3 | `003-objectdatabase-decompress-guard` | `ObjectDatabase.cpp` | `getData()`, `getDataNoTx()`, and `getNextKeyAndValue()` check the `uncompress()` return value. Skip corrupt records instead of processing garbage. |
| 4 | `004-orb-null-check` | `DistributedObjectBroker.cpp` | Null check for `objectManager` in `lookUp()`. Prevents crash without a fully initialized ORB. |

**After building dbconvert**, you can revert these patches since they're only needed at compile time:

```bash
cd /path/to/MMOCoreORB/utils/engine3
git checkout -- .
```

### Step 4: Patch Autogenerated Files

Every autogenerated `readObject()` method calls `initializeTransientMembers()` at the end. During conversion, this corrupts data (resets factions, wipes building access, segfaults on missing managers). This step wraps the call so it only runs during normal server operation.

```bash
python3 /path/to/swgemu-dbconvert/scripts/patch_autogen.py /path/to/MMOCoreORB/src
```

The script takes the path to your `MMOCoreORB/src` directory as a command-line argument. Expected output:

```
Done. Patched: 325, Skipped: 0, Errors: 0
```

The exact count depends on your codebase — anywhere from 300-400 is normal.

**Re-run this script after every `make rebuild-idl`** since IDL regeneration overwrites the patched files.

### Step 5: Edit `src/CMakeLists.txt`

Open `src/CMakeLists.txt` and add three blocks. Here's where each one goes:

#### 5a. Add the executable (after the `add_executable(core3 ...)` line)

Find this line:
```cmake
add_executable(core3 main.cpp server/ServerCore.cpp ...)
```

Add immediately after it:
```cmake
# Create dbconvert binary (standalone database converter)
if(EXISTS "${CMAKE_CURRENT_SOURCE_DIR}/tools/dbconvert.cpp")
add_executable(dbconvert EXCLUDE_FROM_ALL tools/dbconvert.cpp server/ServerCore.cpp ${tre3_sources} ${tre3_headers} ${zone3_sources} ${zone3_headers})
endif()
```

#### 5b. Add dependencies (after `add_dependencies(core3 engine3)`)

Find this line:
```cmake
add_dependencies(core3 engine3)
```

Add immediately after it:
```cmake
if(TARGET dbconvert)
add_dependencies(dbconvert engine3 idlobjects)
endif()
```

#### 5c. Add link libraries (after core3's `target_link_libraries` block)

Find the end of core3's link section — look for the jemalloc conditional and the
`use_gold(core3)` conditional. After those, add:

```cmake
# dbconvert link libraries (same as core3)
if(TARGET dbconvert)
target_link_libraries(dbconvert
    ${whole_archive_start}
    db3 conf3 web3 status3 ping3 login3 chat3 core3terrain core3templates core3pathfinding idlobjects
    ${whole_archive_end}
    engine3
    ${Boost_LIBRARIES}
    ${GCCLIBATOMIC_LIBRARY} ${PTHREAD_LIB_LINK} ${MYSQL_LIB}
    ${LUA_LIBRARIES}
    ${DB_LIBRARIES} ${OPENSSL_LIBRARIES}
    ${ZLIB_LIBRARIES} ${CMAKE_DL_LIBS})

if (NOT ENABLE_ASAN AND NOT ENABLE_TSAN AND NOT ENABLE_UBSAN AND JEMALLOC_FOUND AND ENABLE_JEMALLOC_LINK)
    target_link_libraries(dbconvert ${JEMALLOC_LIBRARIES})
endif()

if(ENABLE_MULTITHREADED_LINKER)
    use_gold(dbconvert)
endif(ENABLE_MULTITHREADED_LINKER)

# Copy dbconvert binary to bin/
add_custom_command(
    TARGET dbconvert POST_BUILD
    COMMAND ${CMAKE_COMMAND} -E copy
    $<TARGET_FILE:dbconvert>
    ${CMAKE_SOURCE_DIR}/bin/)
endif()
```

**Note:** If your server has different library names or additional link dependencies,
adjust accordingly. The dbconvert target needs the same libraries as core3.

The `if(EXISTS)` / `if(TARGET)` guards are optional but recommended — CMake won't error if `dbconvert.cpp` is removed later.

### Step 6: Build

```bash
cd MMOCoreORB/build

# Re-run cmake to pick up the new target
cmake ..

# Build just dbconvert (doesn't rebuild core3)
make dbconvert -j$(nproc)
```

The binary will be copied to `MMOCoreORB/bin/dbconvert`.

**Note:** The first build compiles all zone/object sources for the dbconvert target
(~1000 files). Subsequent builds are incremental and fast.

### Step 7: Convert

**Make sure core3 is NOT running.** The converter and the server cannot access the same
BDB environment simultaneously.

dbconvert must be run from the `bin/` directory where `databases/` and `conf/config.lua` are located. The tool loads `conf/config.lua` (and optionally `conf/config-local.lua`) on startup for BDB environment configuration.

```bash
cd MMOCoreORB/bin

# Convert all databases (classic mode — converts everything)
./dbconvert all

# Or use smart mode (probes per class, skips unchanged — typically 90%+ faster)
./dbconvert all --smart
```

### Checking Errors

```bash
cat databases/dbconvert_errors.log
```

Each error includes the database name, object ID, class name, and error message:

```
[sceneobjects] OID=0x0001000009a75efe class=Component error=loadPersistentObject returned null (unregistered class?)
```

| Error | Meaning | Action |
|-------|---------|--------|
| `loadPersistentObject returned null` | Object's template CRC not registered | Usually custom items from removed content. Safe to ignore. |
| `Could not load database` | BDB file exists but isn't in the manifest | Orphan .db file. Safe to ignore. |
| `Exception during readObject` | Corrupt serialized data | Investigate the specific object. |

Errors do **not** stop conversion. A small number (< 50 out of 1M+) is normal for production databases with legacy custom content.

### Step 8: Start the Server

```bash
./core3
```

The server should boot with zero CRC mismatches — no objects will be marked dirty on first load.

## Commands

### Full Pipeline

```bash
./dbconvert all                 # Phases 1-2-3-4, classic mode
./dbconvert all --smart         # Phases 1-2-3-4, smart mode
```

Runs all four phases in order. If interrupted, re-run the same command — Phase 3 skips already-converted databases via `.converted` manifest files.

### Individual Phases

Each phase can be run independently. Phase gates enforce ordering: each phase requires the previous phase's manifest to exist.

```bash
./dbconvert clean               # Phase 1: Strip LSNs, clean environment
./dbconvert hashfix             # Phase 2: Byte-level hash replacement
./dbconvert hashfix --smart     # Phase 2: Hash replacement + metadata scan
./dbconvert reserialize         # Phase 3: Classic (all records)
./dbconvert reserialize --smart # Phase 3: Smart (probe + selective)
./dbconvert finalize            # Phase 4: Checkpoint + cleanup
```

Running phases individually is useful for:
- **Debugging:** Run just Phase 2 to verify hash fix counts match expectations
- **Incremental work:** Run Phase 1-2, inspect, then decide classic vs smart for Phase 3
- **Recovery:** If Phase 3 fails, fix the issue and re-run Phase 3 alone

### Resume After Interruption

If conversion is interrupted (crash, Ctrl+C, power loss), re-run the same command:

```bash
./dbconvert all --smart
```

Already-converted databases are tracked via `.converted` manifest files and automatically skipped. To force re-conversion of a specific database, delete its manifest and re-run Phase 3:

```bash
rm databases/sceneobjects.converted
./dbconvert reserialize --smart
```

## Troubleshooting

### "Could not open databases/ directory"

Run `dbconvert` from the `bin/` directory where `databases/` is located.

### Phase gate error ("Phase X has not been completed")

Run the required phase first, or use `./dbconvert all` to run the full pipeline.

### Patch script reports "Skipped: 325, Patched: 0"

The pattern wasn't found. Check that:
1. Your autogen files are built (`make rebuild-idl` or `make idlobjects`)
2. The pattern in the script matches your autogen format. Open any autogen .cpp file,
   find the `readObject()` method, and look at how it ends. It should be:
   ```cpp
       stream->setOffset(_currentOffset + _varSize);
       }

       initializeTransientMembers();
   }
   ```
   If the formatting is different, update the `OLD_PATTERN` in `patch_autogen.py`.

### Build fails with undefined references

Make sure you ran `cmake ..` after adding the dbconvert target to CMakeLists.txt.
The `EXCLUDE_FROM_ALL` flag means dbconvert won't appear in `make all` — you must
explicitly build it with `make dbconvert`.

### Conversion crashes with segfault

- Verify the engine3 patches were applied (Step 3). All 4 are required.
- Verify the autogen patch was applied (Step 4). Without it,
  `initializeTransientMembers()` tries to access managers and zones that don't exist.
- Check that `Core::MANAGED_REFERENCE_LOAD` is being set to `false` (this is in the
  dbconvert source and should work out of the box).

### Server boots but objects have wrong data

The autogen patch may not have been applied correctly. Verify with:

```bash
grep -r "MANAGED_REFERENCE_LOAD.*initializeTransientMembers" src/autogen/ | wc -l
```

This should match the number of patched files (300+). If it's 0, re-run the patch script.

### Server boots with "skill points on load mismatch" warnings

This typically means skill definitions changed between your old and new code versions.
It's not a conversion error — the server is recalculating skill points with updated
skill trees. The players' actual skills are preserved; only the point totals differ.
