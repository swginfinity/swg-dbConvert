# SWGEmu Database Converter

A standalone tool for migrating SWGEmu Berkeley DB databases between server code versions.

When SWGEmu server code changes its C++ class definitions, the serialized objects stored in
Berkeley DB become incompatible. This tool performs a safe, offline conversion:

1. **Hash Fix** (Pass 1) - Byte-level repair of renamed serialization field hashes
2. **Reserialize** (Pass 2) - C++ round-trip: load each object with current code, re-serialize,
   write back to BDB
3. **BDB Cleanup** - Dump/reload databases to strip environment dependencies

The tool processes 1M+ objects in ~2 minutes with bounded memory usage (500-object batches
with DOB eviction).

## Why a Standalone Binary?

The conversion **cannot** safely run inside `core3` during normal server startup because:

- The server's CRC dirty-check mechanism (`ObjectManager::deSerializeObject`) compares
  stored bytes against `writeObject()` output. If they differ, the object is marked dirty
  and re-saved on the next save cycle. After a code version change, **every object** would
  mismatch, causing millions of dirty writes that tank the server.

- Running conversion as a standalone tool with `MANAGED_REFERENCE_LOAD=false` and the
  `initializeTransientMembers()` patch (explained below) ensures a clean, one-time
  conversion with zero dirty objects on first boot.

## Quick Start

The build script handles everything automatically — engine3 patches, autogen patches,
CMake configuration, and compilation. Engine3 is left clean after the build.

```bash
# 1. Back up your databases
./scripts/backup.sh pre_convert /path/to/bin/databases

# 2. Build dbconvert (patches engine3 temporarily, reverts when done)
./scripts/build.sh /path/to/MMOCoreORB

# 3. Convert
cd /path/to/MMOCoreORB/bin
./dbconvert all

# 4. Start the server
./core3
```

## Build Process

### Automated (Recommended)

The `scripts/build.sh` script handles the entire build process:

```bash
./scripts/build.sh /path/to/MMOCoreORB
```

What it does:

1. **Copies** `dbconvert.cpp` into the server source tree
2. **Applies** 4 engine3 patches for standalone mode support
3. **Patches** autogenerated `readObject()` methods (via `patch_autogen.py`)
4. **Adds** the dbconvert CMake target to `CMakeLists.txt`
5. **Builds** the dbconvert binary with `cmake .. && make dbconvert`
6. **Reverts** all engine3 patches and CMakeLists.txt changes on exit

Engine3 is **always left clean** — patches are applied only during compilation and
automatically reverted, even if the build fails.

### Manual

If you prefer to apply changes yourself, see the "Required Code Changes" section below.

## Engine3 Patches

The `patches/` directory contains 4 git-format patches for engine3. These are needed
because dbconvert runs as a standalone tool without the full server infrastructure.

| Patch | Purpose |
|-------|---------|
| `001-taskmanager-standalone-mode` | Zero-worker guard — returns early when workerCount < 0, runs tasks inline when no queues exist |
| `002-uncompress-error-handling` | Changes `uncompress()` from void to bool — returns false on Z_DATA_ERROR instead of silently returning corrupt data |
| `003-objectdatabase-decompress-guard` | getData/getDataNoTx/getNextKeyAndValue check uncompress return value — skip corrupt records instead of crashing |
| `004-orb-null-check` | Null check for objectManager in DistributedObjectBroker::lookUp — prevents crash in standalone mode |

These patches do NOT affect core3 behavior — they only add guards for edge cases that
only occur in standalone tool mode (zero workers, no object manager).

## Required Code Changes (Manual Build)

If you prefer not to use `build.sh`, here is every change needed.

### Change 1: Add `dbconvert.cpp`

**File:** `src/tools/dbconvert.cpp` (new file)

Copy `src/tools/dbconvert.cpp` from this repo into your source tree. No modifications needed
unless you have additional hash renames beyond the QuadTreeEntry → TreeEntry change.

The hash table at the top of the file defines the field renames:

```cpp
static const HashReplacement HASH_TABLE[] = {
    { 0x2970c5d9, 0x763502f7, "coordinates" },  // QuadTreeEntry → TreeEntry
    { 0x5a47d37d, 0xb1649cd1, "bounding"    },
    { 0x5284d7c8, 0x256e90b3, "parent"      },
    { 0xac3d85f4, 0xdbd7c28f, "radius"      },
};
```

If your migration involves different field renames, update these entries. The hash values
are the `STRING_HASHCODE()` of the old and new field names from the IDL definitions.

### Change 4: Patch Autogenerated `readObject()` Methods (Critical)

**Files:** ~325 files in `src/autogen/**/*.cpp`

**Why this is needed:**

Every autogenerated `readObject()` method ends with a call to `initializeTransientMembers()`.
During normal server operation, this is correct — it sets up runtime state after loading from
the database.

During **conversion**, however, this call is catastrophic:

- `TangibleObjectImplementation::initializeTransientMembers()` resets `faction = 0` for
  non-standard factions
- `BuildingObjectImplementation::initializeTransientMembers()` clears `registeredPlayerIdList`
  and calls `updatePaidAccessList()`
- `SceneObjectImplementation::initializeTransientMembers()` triggers TemplateManager lookups
  and component creation (which segfaults without a running server)
- `ManagedObjectImplementation::initializeTransientMembers()` resets `lastCRCSave = 0`

The result: converted objects have corrupted data — buildings at (0,0,0), zeroed factions,
cleared access lists, wrong skill points, etc.

**The fix:**

The `scripts/patch_autogen.py` script patches every autogenerated `readObject()` to guard
the call with `Core::MANAGED_REFERENCE_LOAD`:

```
Before:
    stream->setOffset(_currentOffset + _varSize);
    }

    initializeTransientMembers();
}

After:
    stream->setOffset(_currentOffset + _varSize);
    }

    if (Core::MANAGED_REFERENCE_LOAD) initializeTransientMembers();
}
```

`Core::MANAGED_REFERENCE_LOAD` is a static bool that is `true` during normal operation and
`false` during conversion. The `dbconvert` binary sets it to `false` before loading any objects.

**How to apply:**

```bash
# From your MMOCoreORB directory:
python3 scripts/patch_autogen.py
```

Output:
```
Done. Patched: 325, Skipped: 0, Errors: 0
```

**Important:** You must re-run this script after every IDL rebuild (`make rebuild-idl`), as
that regenerates the autogen files and overwrites the patch.

## Architecture Deep Dive

### The CRC Dirty-Mark Problem

The SWGEmu object persistence layer uses a CRC-based dirty detection system:

1. **On load** (`ObjectManager::deSerializeObject`): Computes CRC32 of the raw BDB bytes
   and stores it as `lastCRCSave` on the object.

2. **On save** (`DOBObjectManager::commitUpdatePersistentObjectToDB`): Calls `writeObject()`
   to serialize the object, computes CRC32 of the result, and compares with `lastCRCSave`.
   If they differ, the object is written to BDB.

When code changes (new fields, renamed fields, different defaults), `writeObject()` produces
different bytes than what's stored in BDB. This means **every object** appears dirty on first
boot after a code change, causing millions of BDB writes that overwhelm the server.

The solution: run `dbconvert` offline to re-serialize all objects. After conversion, the
stored bytes match exactly what `writeObject()` would produce, so `lastCRCSave` matches
and zero objects are dirty on first boot.

### The QuadTreeEntry → TreeEntry Hash Fix

SWGEmu's serialization format is flat, hash-based:

```
[uint16 varCount]
[uint32 nameHash][uint32 dataSize][N bytes data]  // field 1
[uint32 nameHash][uint32 dataSize][N bytes data]  // field 2
...
```

Field names are stored as 32-bit hashes (`STRING_HASHCODE`). When `QuadTreeEntry` was
renamed to `TreeEntry` upstream, four field hashes changed:

| Field       | Old Hash (QuadTreeEntry) | New Hash (TreeEntry) |
|-------------|--------------------------|----------------------|
| coordinates | `0x2970c5d9`             | `0x763502f7`         |
| bounding    | `0x5a47d37d`             | `0xb1649cd1`         |
| parent      | `0x5284d7c8`             | `0x256e90b3`         |
| radius      | `0xac3d85f4`             | `0xdbd7c28f`         |

Pass 1 of the converter scans the raw bytes of every record and replaces old hashes with
new ones. This is done without creating any C++ objects — it's a pure byte-level operation
that runs at ~500K records/second.

### The Reserialize Round-Trip

Pass 2 loads each object through the full C++ deserialization chain
(`ObjectManager::loadPersistentObject`), then re-serializes it with `writeObject()` and
writes the result directly to BDB with `putData()`.

Key safety measures:

- **`Core::MANAGED_REFERENCE_LOAD = false`**: Prevents ManagedReference from resolving
  cross-references (which would trigger loading dependent objects and blow up memory)
- **`initializeTransientMembers()` suppressed**: The autogen patch prevents runtime
  initialization from corrupting persistent data during conversion
- **500-object batches**: Objects are evicted from the Distributed Object Broker after
  each batch to keep memory bounded. This allows processing 63M+ records.
- **Direct `putData()`**: Writes go straight to BDB, bypassing the dirty-flag system
  that would queue objects for async save

### Error Handling

The converter logs errors at two levels:

- **stdout**: Summary counts (success) + per-error detail (class name, OID, database,
  exception message)
- **`databases/dbconvert_errors.log`**: Persistent error log with the same detail, plus
  per-database summaries and timestamps

Common errors:
- "loadPersistentObject returned null (unregistered class?)" — The object's `serverObjectCRC`
  doesn't match any registered template. Usually custom items whose templates were removed.
- "Could not load database" — The `.db` file exists but has no registered name in the
  databases manifest. Harmless.

### Resume Support

Each successfully converted database gets a `.converted` manifest file in `databases/`.
If `dbconvert` is interrupted, re-running `./dbconvert all` skips already-converted
databases automatically.

## Adapting for Your Server

### Different Source/Target Versions

The hash table in `dbconvert.cpp` is specific to the QuadTreeEntry → TreeEntry rename.
If your migration involves different field renames:

1. Identify which IDL fields changed names between your versions
2. Compute `STRING_HASHCODE("OldClassName.fieldName")` and
   `STRING_HASHCODE("NewClassName.fieldName")` for each
3. Add entries to the `HASH_TABLE[]` array

If no field names changed (just new fields added), you can set `hashOnlyMode` or simply
skip Pass 1 — Pass 2 (reserialize) will handle new/removed fields automatically.

### Custom Object Types

If your server has custom C++ object types that aren't in the standard SWGEmu codebase,
they'll be deserialized and re-serialized automatically as long as:

1. Their class is registered in `ObjectManager`
2. Their `readObject()` and `writeObject()` methods work correctly
3. Their autogenerated files have been patched (Change 4 above)

Objects with unregistered CRCs will be logged as errors but won't stop the conversion.
